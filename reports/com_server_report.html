3. COM Architecture Overview

The AVImark COM Server follows the standard Microsoft COM (Component Object Model) architecture. It is an in-process server (DLL-based) that runs within the calling process's address space. Each interface is an IDispatch automation interface, meaning it supports late-binding from scripting languages (VBScript, PowerShell, Python via win32com).

Access Model

Authentication: The IAVIFileInProc interface includes a Login(Password) method, indicating the COM server requires authentication before allowing data access. The password is likely the same as the AVImark server password found in AVImarkServer.ini (stored as a hex-encoded hash).

Communication Pattern

All interfaces use a handle-based navigation pattern:

  1. Search — GetFirstByName(key) returns a handle (record pointer)
  2. Navigate — GetNextByName(key) iterates through results
  3. Read — GetMoreStuff(handle) returns FieldNames + FieldValues
  4. Write — SetFieldValues() or SetFieldValuesByName() modifies data
  5. Create — CreateInvoice(), CreateOrder(), AddNewEntry(), AddRecordEx()

6. Security Analysis

Authentication

  • IAVIFileInProc.Login(Password) requires authentication before raw file access
  • Password is likely the server password from AVImarkServer.ini (hex-encoded hash)
  • Other interfaces may or may not require prior Login — needs testing

Access Control

  • COM server is 32-bit InprocServer32 — runs in the calling process's context
  • No apparent role-based access control within the COM API
  • Once authenticated via Login(), appears to have full access to all operations
  • The server process (AVImarkServer.exe) runs as LocalSystem — high privilege

Risk Assessment

RiskSeverityMitigation
Write operations could corrupt dataHIGHAlways test on clone server first. Backup before write operations.
No transaction/rollback mechanism visibleHIGHVerify each write with subsequent read. Implement application-level logging.
Password may be extractable from INIMEDIUMRestrict file access to AVImark directory. Use OS-level permissions.
COM server could crash AVImark if misusedMEDIUMIn-process COM runs in caller's space, not server's. But shared files could deadlock.

These use SOAP/XML and could be leveraged for distributor-integrated automated ordering.

8. Automation Use Cases

🛒 Automated Inventory Ordering

1. Supabase analytics detects low inventory (PO history + usage patterns)
2. System generates suggested order list
3. Staff approves (or auto-approve for routine items)
4. COM: CreateOrder("IDEXX") → AddItem("CBC") → AddItem("CHEM17") → ...
5. Order appears in AVImark's PO system
6. Distributor receives order via eOrdering SOAP integration

📋 AI SOAP Notes

1. Vet completes visit (services billed, prescriptions written)

3. AI generates SOAP note from structured data + voice notes

9. Registration Status & Next Steps

Current Status

Registration: FAILED (Silent)
regsvr32 "C:\AVImark\AVImarkCOMServer.dll" completes without error code but does not write ProgIDs or CLSIDs to the Windows registry. The 32-bit variant (C:\Windows\SysWOW64\regsvr32.exe) hangs with a dialog box (inaccessible on headless server).

Possible Causes

  1. Runtime dependency: DLL may require AVImark client application context (not just the server service)
  2. Missing Delphi runtime packages: Large Delphi COM servers sometimes need BPL packages
  3. Server vs. Workstation: May need to be registered from an AVImark workstation, not the server
  4. License check: DllRegisterServer may verify AVImark license before registering
  5. Path dependency: May need to run regsvr32 from C:\AVImark directory
